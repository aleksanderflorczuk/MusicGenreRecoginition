# Assignment 6 - Feature Engineering and Preprocessing

## Objective and relation to previous analysis

The objective of this stage is to formalize the preprocessing and feature engineering strategy for the Music Genre Recognition project. The project uses audio files from ten music genres and represents each recording as a fixed-length numerical feature vector extracted with `librosa`. The target variable is `genre`, while the input variables describe tempo, energy, spectral characteristics, MFCC coefficients, and chroma features.

This assignment continues the exploratory analysis performed in Assignment 5, documented in `PDF/EDA_s27833_2026.pdf` and implemented in `src/eda.py`. The EDA showed that the dataset contains 1000 observations and 62 columns: 61 numerical predictors and one categorical target variable. The class distribution is balanced, with 100 tracks for each genre: blues, classical, country, disco, hiphop, jazz, metal, pop, reggae, and rock. Therefore, no class balancing procedure is planned at the preprocessing stage.

The EDA also showed several issues that must be addressed before modeling:

- Features are measured on very different numerical scales. For example, `tempo` is expressed in beats per minute, spectral features are measured in frequency-related units, while MFCC and chroma features are unitless coefficients. This makes scaling necessary for distance- and margin-based models such as SVM.
- Some global spectral variables are strongly correlated. For example, `spectral_centroid_mean` and `spectral_rolloff_mean` are almost linearly related. This does not invalidate the dataset, but it must be considered when interpreting models and when selecting features.
- The dataset does not contain missing values. Therefore, imputation is not required for the current extracted feature table.
- The dataset contains duplicate rows or duplicate feature vectors. These observations must be investigated because they may represent repeated audio fragments or feature-equivalent recordings.
- PCA visualization from Assignment 5 suggested partial genre separability, especially for some genres, but also visible overlap between genres. This means preprocessing should preserve informative continuous variation instead of discretizing features.

The raw audio files and the extracted feature table remain conceptually separate. The raw audio data are the source data. The processed dataset is a reproducible numerical representation generated by the feature extraction script. From Assignment 6 onward, model comparison should be performed using a fixed preprocessing pipeline unless a change is explicitly documented and justified.

## A. Preprocessing Strategy Overview

The planned preprocessing pipeline consists of the following stages:

1. Audio feature extraction from each `.wav` file.
2. Construction of a tabular dataset with one row per track and one column per extracted feature.
3. Data validation and cleaning.
4. Stratified train-test split.
5. Model-specific preprocessing fitted only on the training data.
6. Cross-validation and hyperparameter tuning using a single integrated pipeline.
7. Final evaluation on the untouched test set.

The feature extraction stage is deterministic and applies the same transformations to all audio files. For each track, the following feature groups are extracted:

- `tempo`
- zero crossing rate mean and standard deviation
- RMS energy mean and standard deviation
- spectral centroid mean and standard deviation
- spectral bandwidth mean and standard deviation
- spectral rolloff mean and standard deviation
- 13 MFCC means and 13 MFCC standard deviations
- 12 chroma means and 12 chroma standard deviations

These feature groups were selected because they describe complementary properties of music recordings. MFCC features capture timbral information, chroma features describe pitch class distribution, RMS describes energy, zero crossing rate captures signal noisiness and high-frequency content, while spectral centroid, bandwidth, and rolloff describe the distribution of energy across the frequency spectrum.

The components that should remain constant across all models are:

- the raw audio source;
- the extracted feature definitions;
- the train-test split procedure;
- the leakage prevention rules;
- the evaluation protocol;
- the target variable definition.

Only model-specific transformations should vary. For example, SVM and logistic regression require scaling, while tree-based models such as random forest can be trained without scaling. However, for fair experimental comparison, the same raw extracted feature table must be used as the input to all model pipelines.

The current repository contains a globally scaled file, `features_extended_scaled.csv`, created during EDA. This file is useful for exploratory visualization, but it should not be used as the primary experimental dataset for model evaluation because global scaling before train-test splitting can leak information from the test set into the training process. The correct experimental approach is to load `features_extended.csv`, split the data, and fit preprocessing only on the training fold or training split.

## B. Data Cleaning and Missing Value Strategy

### Missing values

The extracted dataset currently contains no missing values. This means that no imputation is required for the present version of the project. This decision is empirical, not convenience-based: the absence of missing values was verified during EDA using column-wise missing value counts.

If missing values appear in later versions of the dataset, the strategy will depend on the reason for missingness:

- If a feature cannot be computed because of a corrupted or unreadable audio file, the preferred strategy is to exclude the affected file and document the exclusion.
- If a small number of numerical values are missing because of numerical instability in feature extraction, median imputation may be used within the training fold.
- If missingness is systematic for a feature group, the feature extraction procedure must be reviewed before modeling.

Median imputation would be preferred over mean imputation because some audio features are skewed and contain outliers. However, since the current dataset has no missing values, adding an imputer is optional. If included for robustness, it must be placed inside the scikit-learn pipeline so that imputation parameters are estimated only from training data.

### Duplicates

EDA detected duplicate rows and duplicate feature vectors. This requires controlled handling because duplicates can bias evaluation if near-identical observations appear in both training and test sets.

The planned strategy is:

1. Identify exact duplicate rows in the extracted feature table.
2. Check whether duplicated feature vectors have the same target label.
3. Remove exact duplicate observations before splitting if they represent repeated entries of the same track.
4. If duplicate feature vectors correspond to different labels, inspect the source files and treat the case as a data quality issue.

Removing duplicates improves generalization assessment because the test set should contain independent examples rather than repeated records already seen during training. The alternative strategy would be to keep duplicates, but this could inflate accuracy by making the test set easier. Therefore, duplicate removal is methodologically safer if the duplicates are confirmed to be repeated records.

### Inconsistent entries

The target label is derived from the genre folder name. Therefore, the allowed genre values are fixed to the ten expected classes. During data validation, labels should be checked against this controlled list. Any unexpected label should be treated as an extraction or directory-structure error.

Numerical feature columns should also be validated. All predictor columns should be numeric, finite, and non-constant. The current dataset contains 61 numerical columns and no constant feature columns. Values such as negative spectral frequencies or non-finite coefficients would indicate an extraction error and should be handled before modeling.

## C. Scaling, Normalization, and Encoding

### Scaling

Standardization is the main scaling strategy for this project. Each numerical feature is transformed by subtracting the training mean and dividing by the training standard deviation:

```text
z = (x - mean_train) / std_train
```

Standardization is appropriate because most models planned for baseline comparison operate on numerical feature vectors and may be sensitive to scale. This is especially important for:

- SVM with RBF kernel, because distances in feature space directly affect the kernel value;
- k-nearest neighbors, if used, because Euclidean distances are scale-dependent;
- logistic regression, because regularization strength is affected by feature scale;
- PCA or other dimensionality reduction methods, because unscaled high-variance features would dominate the components.

Tree-based models such as decision trees, random forest, and gradient boosting do not require scaling because their split criteria are invariant to monotonic transformations of individual features. Nevertheless, scaling can still be included in model-specific pipelines when needed. For fair comparison, each model should receive the preprocessing appropriate to its assumptions, but all models must be evaluated using the same train-test split and cross-validation protocol.

Scaling must be applied after the train-test split and within cross-validation. The scaler must be fitted only on the training data in each fold. The test set and validation folds must be transformed using parameters learned from the corresponding training data. This prevents data leakage.

The file `features_extended_scaled.csv` should therefore be treated as an exploratory artifact, not as the final modeling input. The correct implementation should use `features_extended.csv` and a pipeline such as:

```python
Pipeline([
    ("scaler", StandardScaler()),
    ("model", SVC())
])
```

When combined with `GridSearchCV`, this structure ensures that standardization is recalculated separately inside every training fold.

### Normalization

Min-max normalization is not selected as the default strategy because several audio features have outliers and different distribution shapes. Min-max scaling can be strongly affected by extreme values, especially in spectral and energy-related variables. Standardization is more appropriate for SVM and linear models in this context.

Robust scaling is an alternative if later experiments show that outliers strongly affect performance. However, switching from standardization to robust scaling would be treated as an experimental variant and documented as a formal pipeline revision.

### Encoding

There are no categorical input features in the current feature table. The only categorical column is the target variable `genre`. Therefore, no one-hot, ordinal, or target encoding is required for predictors.

The target labels can remain as strings because scikit-learn classifiers can internally handle string class labels. If a model requires numeric labels, label encoding may be applied to the target only. This does not create leakage because the encoding maps class names to fixed identifiers and does not use predictor distributions.

High-cardinality categorical handling is not applicable because no high-cardinality categorical predictors exist in the current dataset.

## D. Feature Engineering and Transformation

The main feature engineering step is the transformation from raw audio waveforms into a structured numerical representation. This is not a simple cleaning step; it defines the representation used by all later models.

### Existing engineered features

The current feature extraction script constructs the following feature groups:

- Tempo: expected to capture rhythmic speed and support separation of genres with different rhythmic profiles.
- RMS energy mean and standard deviation: expected to distinguish calmer genres from louder or more dynamic genres.
- Zero crossing rate mean and standard deviation: expected to describe noisiness and high-frequency signal behavior.
- Spectral centroid, bandwidth, and rolloff statistics: expected to capture brightness and spectral spread.
- MFCC means and standard deviations: expected to represent timbre, which is central in music genre classification.
- Chroma means and standard deviations: expected to capture harmonic and pitch-class information.

Each time-varying audio descriptor is summarized using mean and standard deviation. This produces a fixed-length vector for each recording, allowing classical machine learning models to be used. The advantage is simplicity and reproducibility. The limitation is that temporal structure inside a track is compressed, so the model cannot directly learn sequence patterns.

### Potential additional features

At this stage, no additional feature group should be added without formal revision, because Assignment 6 fixes the preprocessing regime for baseline modeling. However, possible future extensions include:

- delta and delta-delta MFCC features to represent short-term temporal change;
- beat-synchronous chroma statistics;
- spectral contrast;
- harmonic-percussive source separation features.

These extensions could improve performance, but they also increase dimensionality and risk overfitting. They should therefore be compared experimentally only after a stable baseline has been established.

### Dimensionality reduction

PCA was used in Assignment 5 for visualization. It should not be used as a mandatory preprocessing step for baseline models unless it is included inside the cross-validation pipeline. PCA fitted on the full dataset before splitting would leak information from the validation and test sets.

For baseline experiments, dimensionality reduction is not selected as the default. The dataset has 61 numerical predictors and 1000 observations, which is manageable for the planned classical models. Keeping the original features also improves interpretability because each feature still corresponds to a known audio descriptor.

PCA may be evaluated later as an experimental variant, especially for SVM or k-nearest neighbors. If used, the number of components must be selected inside cross-validation and documented.

### Risks

The main risks of the current feature design are:

- Multicollinearity among spectral features, which can reduce interpretability for linear models.
- Overfitting if too many engineered features are added without sufficient validation.
- Leakage if scaling, PCA, imputation, or feature selection are fitted before train-test splitting.
- Loss of temporal information because each track is summarized by global statistics.

These risks are controlled by using integrated pipelines, keeping the baseline feature set fixed, and evaluating transformations only inside the training folds.

## E. Feature Selection Strategy

Feature selection is optional for the first controlled baseline. The initial baseline should use all 61 extracted numerical features because they were defined before model training and are justified by audio-domain relevance.

A secondary experiment may use feature selection, but only under strict validation rules. The currently used manual subset in `train.py` should be treated as an exploratory selection, not as final methodology, unless the selection criteria are explicitly documented and reproduced.

If feature selection is applied, the recommended approach is:

- Method: filter or embedded selection.
- Filter criteria: correlation analysis, mutual information, or ANOVA F-score.
- Embedded criteria: regularized linear model or tree-based feature importance.
- Validation protocol: selection fitted only inside training folds as part of a scikit-learn pipeline.

A valid feature-selection pipeline may look like:

```python
Pipeline([
    ("scaler", StandardScaler()),
    ("selector", SelectKBest(score_func=f_classif, k=30)),
    ("model", SVC(kernel="rbf"))
])
```

The value of `k` must be selected using cross-validation. The selector cannot be fitted on the full dataset before cross-validation because that would allow information from validation folds to influence which features are retained.

Feature selection may improve computational cost and reduce noise, but it can also reduce interpretability if selected subsets vary across folds. For the thesis, both the selected features and the selection rule must be reported. If interpretability is a priority, domain-based grouping and correlation analysis may be preferable to purely automatic selection.

## F. Pipeline Formalization and Reproducibility

The preprocessing is implemented as reproducible code rather than as manual spreadsheet-style transformations. The feature extraction stage is implemented in `src/extract_features.py`, the SVM training pipeline is implemented in `src/train_classifier.py`, and the baseline comparison pipeline is implemented in `src/baseline_models.py`. The raw audio location is configured in `config.py`, while the canonical extracted feature table is `data/features_extended.csv`.

The core modeling structure is a scikit-learn `Pipeline`, which keeps learned preprocessing and model fitting together:

```python
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler
from sklearn.model_selection import GridSearchCV, train_test_split
from sklearn.svm import SVC

X = df.drop(columns="genre")
y = df["genre"]

X_train, X_test, y_train, y_test = train_test_split(
    X,
    y,
    test_size=0.2,
    stratify=y,
    random_state=30
)

pipeline = Pipeline([
    ("scaler", StandardScaler()),
    ("model", SVC(kernel="rbf"))
])

param_grid = {
    "model__C": [1, 10, 50],
    "model__gamma": ["scale", 0.01, 0.001]
}

grid = GridSearchCV(
    pipeline,
    param_grid=param_grid,
    cv=5,
    scoring="accuracy",
    n_jobs=-1
)

grid.fit(X_train, y_train)
```

This formalization separates training and inference transformations. During training, the scaler is fitted on the training data. During inference, the same scaler parameters are reused to transform new observations before prediction. No manual scaling or feature selection should occur between training runs.

The repository uses explicit configuration values for reproducibility:

| Element | Current value or path |
|---|---|
| Raw audio path | `config.py`, variable `AUDIO_DATASET_PATH` |
| Feature extraction code | `src/extract_features.py` |
| Canonical modeling data | `data/features_extended.csv` |
| Exploratory scaled data, not used for final modeling | `data/features_extended_scaled.csv` |
| SVM pipeline code | `src/train_classifier.py` |
| Baseline comparison code | `src/baseline_models.py` |
| Target column | `genre` |
| Number of predictors | 61 numerical audio features |
| Duplicate policy | `drop_duplicates()` before splitting |
| Missing-value policy | raise an error if missing values are present |
| Test size | `0.2` |
| Split type | stratified train-test split |
| Random seed | `30` |
| Cross-validation | 5-fold CV in `GridSearchCV`; stratified 5-fold CV in `src/baseline_models.py` |
| Scaling | `StandardScaler` inside `Pipeline` |
| Main metric | accuracy |
| Supporting baseline metrics | macro precision, macro recall, macro F1 |

The deterministic run commands from the project root are:

```powershell
python src/extract_features.py
python src/train_classifier.py
python src/baseline_models.py
```

The first command regenerates `data/features_extended.csv` from the fixed raw audio directory. The training commands use fixed split parameters and fixed model random states where applicable. No manual editing of the feature table is required between runs.

For later inference, the fitted pipeline should be saved and reused as a single object. This means that a new audio file must first be converted into the same 61-feature schema, and then the saved pipeline should apply the already learned scaler parameters before prediction. The inference path must not refit the scaler, selector, PCA transformer, or model on new examples.

The reproducible data flow is:

```text
raw audio files
-> deterministic feature extraction
-> features_extended.csv
-> duplicate and missing-value validation
-> train-test split
-> pipeline fitted on training data only
-> cross-validation and hyperparameter tuning
-> final evaluation on test set
-> saved pipeline for inference
```

This structure prevents leakage because the test set remains untouched until final evaluation. It also ensures that all models are compared under the same experimental conditions.

## G. Impact on Experimental Design

Preprocessing choices directly affect model comparability. If one model is evaluated on globally scaled data and another is evaluated using fold-specific scaling, the comparison is not fair. Therefore, all baseline models in Assignment 7 should use the same raw extracted feature table and the same splitting procedure. Differences between models should come from model architecture and explicitly defined model-specific preprocessing, not from hidden transformations.

The default preprocessing strategy for baseline modeling is:

- use `features_extended.csv` as input;
- remove confirmed duplicate records before splitting;
- keep all 61 numerical predictors for the first baseline;
- use stratified train-test split;
- apply `StandardScaler` inside the pipeline for scale-sensitive models;
- avoid global preprocessing before cross-validation;
- keep the final test set untouched until the final evaluation.

Alternative preprocessing strategies may be compared experimentally after the baseline is established. Reasonable alternatives include robust scaling, feature selection, PCA, or additional audio descriptors. Each alternative must be implemented as a separate pipeline variant and evaluated using the same validation protocol.

These preprocessing decisions support the research question by creating a controlled basis for comparing machine learning models for music genre classification. The goal is not only to maximize accuracy but also to ensure that the reported results reflect genuine model performance rather than evaluation bias introduced by preprocessing.

From this point forward, the preprocessing pipeline is considered fixed for baseline experiments. Any modification, such as adding new features, changing the scaler, removing feature groups, or introducing dimensionality reduction, must be documented as a formal revision and justified empirically.
